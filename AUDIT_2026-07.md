# Full codebase audit — July 2026

Read-only review. No code was changed as part of this audit. Scope: `src/`, `api/`, `scripts/` (the latter included for hygiene only, excluded from performance scoring). Findings are ranked **blocker / should-fix / nice-to-have**, then by risk-adjusted value within each tier. Every finding lists file references, one-line evidence, blast radius, and an explicit call-out wherever fixing it risks changing user-visible behaviour.

Conducted as six parallel, systematic passes (architecture/duplication, maintainability/dead-code, performance/bundle, memory leaks, data-flow/API/states, accessibility/responsive), each grepping broadly across the whole tree rather than sampling. Where two passes independently found the same issue, that's noted — it's corroboration, not double-counting.

## Blockers

### B1. Entire app ships as one 5.0 MB JS bundle (1.39 MB gzipped) — zero code-splitting
- **File(s)**: `vite.config.js` (no `manualChunks`, no route splitting), `src/App.jsx` (57 of 115 pages statically imported), `src/components/guests/ImportGuestModal.jsx:2` (`xlsx`), `src/components/schedule/WeddingDayTimelineBuilder.jsx:3-4` + `src/lib/assetExport.js:23-24` (`jspdf`, `html2canvas`)
- **Evidence**: `npm run build` → `dist/assets/index-*.js` = 5.0 MB raw / 1.39 MB gzipped, single chunk. Zero `React.lazy`/`lazy(` hits anywhere in `src/`. xlsx/jspdf/html2canvas (each 200-400KB+) load on every page, including the marketing homepage, despite being reachable from only one or two narrow flows (bulk guest import, PDF/schedule export).
- **Blast radius**: Single biggest lever in the whole audit. Every visitor — including anonymous marketing-page visitors — downloads the entire authenticated app's code and three heavy libraries they'll likely never touch.
- **Behaviour-change risk**: LOW if done right (lazy-loaded routes render identically once loaded), but requires a deliberate `<Suspense>` fallback design — a missing/wrong fallback shows a blank flash on slow connections. Converting the three heavy-lib imports to dynamic `import()` inside their click handlers is safe since those call sites are already async.

### B2. Primary sidebar navigation has zero keyboard support
- **File(s)**: `src/components/layout/AnimatedSidebar.jsx:176-224` (`NavItem`, used by every nav row in both desktop sidebar ~277-389 and `MobileSidebarContent` ~461-618)
- **Evidence**: Every nav item is a bare `<div onClick={...}>` — no `role="button"`, `tabIndex`, or `onKeyDown`. 11 onClick divs in this one file alone, more than any other file in the repo.
- **Blast radius**: This single component renders every nav item (Dashboard, Guests, Budget, Seating, Studio, Help, Collaborate, etc.) on every authenticated page. A keyboard-only or screen-reader user cannot navigate the app at all today.
- **Behaviour-change risk**: LOW — adding `role="button"`, `tabIndex={0}`, and an Enter/Space `onKeyDown` is purely additive, same click semantics.

### B3. Global chrome controls (notification bell, mobile menu, Ava toggle) have no accessible name
- **File(s)**: `src/Layout.jsx:146` (bell), `:467` (raw inline hamburger `<svg>`), `:499` (Ava chat toggle)
- **Evidence**: None of the three icon-only `<button>`s has `aria-label`/visible text. Only 18 `aria-label` hits exist in the entire codebase, none in `Layout.jsx`.
- **Blast radius**: `Layout.jsx` wraps every authenticated page — a screen-reader user cannot identify "menu," "notifications," or "chat with Ava" anywhere in the product.
- **Behaviour-change risk**: LOW — pure addition.

### B4. 78 confirmed dead component files (~4,200+ lines), one recurring pattern
- **File(s)**: Full list across `src/components/` — largest chains: 7 orphaned "old AI-assistant" components (`AIStylingAssistant.jsx` 1200 lines, `AIBudgetAssistant.jsx` 737, `AIBudgetInsights.jsx` 604, `AIVendorAssistant.jsx` 589, `AITimelineGenerator.jsx` 495, `AIGuestResponseAssistant.jsx` 355, `dashboard/AIWeddingAssistant.jsx` 229 — superseded by the unified Ava system and the still-live `shared/AIWeddingAssistant.jsx`); the entire `src/components/website-editor/` directory (4 files, a false-positive-checked coincidence with the *route path string* `/website-editor`, which actually renders `StudioWebsite.jsx`); old invitation builder subsystem (5 files); old seating-assignment components (3); old Design-Studio tab set (5); 2 stray guest-website pages; `DailyBriefing.jsx` (443 lines, a near-duplicate of the live `DailyUpdate.jsx` — flagged independently by both the architecture and maintainability passes); 9 unused motion/animation helpers; ~40 more single-file orphans (full list in the maintainability pass transcript, available on request) — plus 21 unused shadcn/ui scaffold primitives, counted separately as lower-confidence/lower-priority.
- **Evidence**: Each verified two ways — zero regex hits anywhere in `src/`/`api/`/`scripts/`, *and* manual confirmation against the actual live parent page's real import list (to rule out dynamic imports, string-based routes, or naming collisions — several near-miss false positives were caught and excluded this way, e.g. a local variable also named `SectionManager`, and a route-string coincidence for `website-editor`).
- **Blast radius**: The single largest cleanup opportunity in the audit — zero interdependencies found among any of the 78 files or with live code.
- **Behaviour-change risk**: **None** — every file confirmed to have zero importers by two independent methods. This is a blocker for cleanup *priority* (given its size), not because it's risky.

## Should-fix

### S1. Checkout silently fails in the in-app upgrade path (revenue-critical)
- **File(s)**: `src/pages/PlanSelection.jsx:46-69` vs. `src/pages/Pricing.jsx:103-169` (same endpoint, same payload, already fixed there)
- **Evidence**: `PlanSelection.jsx` only checks `data.url`, never `res.ok`/`data.error`, and swallows all failures in an empty `catch {}` — button just stops spinning with zero feedback. `Pricing.jsx`'s near-identical `startCheckout` already sets a distinct, user-visible `checkoutError` per failure mode.
- **Blast radius**: `PlanSelection.jsx` is reachable from `Account.jsx` and `CouplesStudio.jsx` as the in-app upgrade path.
- **Behaviour-change risk**: LOW — porting the already-shipped `Pricing.jsx` pattern is additive only.

### S2. Dashboard.jsx swallows fetch failures into an indistinguishable "empty" state
- **File(s)**: `src/pages/Dashboard.jsx:133-161`
- **Evidence**: Bare empty `catch {}` around every data load; contrast with `Vendors.jsx`/`Seating.jsx`, which correctly `toast.error(...)` on the identical failure class.
- **Blast radius**: Dashboard is the first page after login — a transient failure looks identical to a legitimately fresh account.
- **Behaviour-change risk**: LOW — additive, mirrors an already-proven pattern in the same codebase.

### S3. No effective caching layer despite React Query being installed
- **File(s)**: `src/lib/query-client.js`, `src/App.jsx` (`QueryClientProvider` wraps the app), `src/lib/resolveMyWedding.js`
- **Evidence**: `useQuery(` appears in only 4 files; 47 page files call `base44.entities.*` directly, and `resolveMyWedding.js`'s helpers do a fresh network round trip on every call with no memoization anywhere in the stack.
- **Blast radius**: Systemic — touches most of the ~80-page authenticated app.
- **Behaviour-change risk**: LOW-MEDIUM. `resolveMyWedding.js`'s own header comment explicitly requires identity to always be re-resolved fresh (never cached across a user switch) — any fix must key the cache to session/user id and invalidate on auth change, not cache blindly.

### S4. Layout.jsx refetches up to 7 requests' worth of data on every single navigation
- **File(s)**: `src/Layout.jsx:263-293`
- **Evidence**: `useEffect(() => fetchData(), [fetchData, location.pathname])` — fires `base44.auth.me()` plus 3 more resolver calls (each of which internally re-calls `auth.me()` too) on every route change, app-wide.
- **Blast radius**: Every navigation across ~80+ pages pays this cost, on top of whatever the destination page itself fetches. A `weddingDetailsSaved` custom-event listener already exists as an explicit invalidation hook, suggesting the `location.pathname` trigger may be redundant.
- **Behaviour-change risk**: MEDIUM — removing `location.pathname` from the dependency array changes *when* data refreshes; the existing event-based invalidation likely covers the real "data changed" case, but this needs a deliberate decision, not a blind removal.

### S5. `useCollaboratorContext()` fetches the same endpoint twice per page load
- **File(s)**: `src/lib/collaboratorContext.js` (no cache/dedupe) + `src/Layout.jsx:260` + 15 page files independently calling it
- **Evidence**: Both `Layout.jsx` and whatever page is rendering call this hook independently — a plain custom hook, not a shared Context — so a single collaborator-session page load fires `/api/collaborator-context` twice concurrently.
- **Blast radius**: Doubles this endpoint's traffic (and its server-side Base44 admin-key round trip) on every collaborator page load.
- **Behaviour-change risk**: LOW — lifting the hook into a single Context provider mounted once in `Layout.jsx` is a wiring change; response shape is unchanged.

### S6. `api/places.js` and `api/places-search.js` handle identical failures with opposite status-code contracts
- **File(s)**: `api/places.js:75-96` vs. `api/places-search.js:55-58,72-75`
- **Evidence**: `places.js` returns `200 { places: MOCK_PLACES, mock: true }` on a real upstream error — silently masking failure as success. `places-search.js` correctly returns `500` for the same failure class.
- **Blast radius**: Callers of `places.js` can't distinguish "this is deliberately the no-API-key mock" from "the live Google API is down right now" without checking undocumented `mock`/`apiError` fields.
- **Behaviour-change risk**: **HIGH if fixed carelessly** — the mock-on-no-key path is a deliberate, documented dev/demo feature ("Returns mock data when GOOGLE_PLACES_API_KEY is not set"). A fix must preserve that path and only change the *upstream-error*/*network-exception* branches to match `places-search.js`. Flag explicitly before touching — a client may already depend on always getting 200 from this specific route.

### S7. Budget/Schedule/Vendors flash a false "empty" state before the initial fetch resolves
- **File(s)**: `src/pages/Budget.jsx` → `BudgetList.jsx:38`; `src/pages/Schedule.jsx` → `ScheduleTimeline.jsx:35`/`ScheduleList.jsx:40`; `src/pages/Vendors.jsx:245`
- **Evidence**: All three gate the empty-state message purely on `items.length === 0`, with no `loading` prop threaded through — real, visible bug (briefly implies data loss) on every load/reload. `Guests.jsx`/`GuestList.jsx` already does this correctly (skeleton while `loading`, empty-state only once `!loading && length === 0`) — proving the right pattern exists in the same codebase.
- **Blast radius**: Every couple's first load and every reload of 3 major pages.
- **Behaviour-change risk**: LOW — additive, directly modeled on the already-shipped `GuestList` fix.

### S8. Guests/GuestList re-renders the entire table on every keystroke or checkbox click
- **File(s)**: `src/pages/Guests.jsx:372` (`filteredGuests` not memoized, unlike sibling `stats`/`guestRoles` in the same file), `:228-485` (all handlers unmemoized), `src/components/guests/GuestList.jsx` (1003 lines, zero `useMemo`/`useCallback`/`memo` anywhere)
- **Evidence**: `selectedIds` (a `Set`) recreated on every toggle; no row-level memoization on a table that can hold hundreds of guests.
- **Blast radius**: For a realistic 150-300 guest wedding, every search keystroke and every checkbox toggle re-renders and re-diffs the entire table, not just the affected row.
- **Behaviour-change risk**: LOW for correctness if dependency arrays are set right, but real risk of stale-closure bugs if not — needs careful testing of inline-edit, bulk-select, and quick-add flows after the change. Not a one-line fix.

### S9. Sequential (non-parallel) Base44 fetches per collaborator page load
- **File(s)**: `api/collaborator-data.js:100-104`
- **Evidence**: `for (const entity of pageConfig.entities) { await adminFetch(...) }` — Seating/Registry/Music each have 3 entities awaited one at a time instead of via `Promise.all`.
- **Blast radius**: ~3x network latency for every collaborator viewing those pages.
- **Behaviour-change risk**: LOW — response shape unchanged, no ordering dependency between the entities.

### S10. Photo/moodboard grids and marketplace thumbnails load eagerly
- **File(s)**: `src/pages/PhotoGallery.jsx:366-372`, `src/components/moodboard/MoodboardGrid.jsx:51,178`
- **Evidence**: No `loading="lazy"` on grid `<img>` tags that can hold arbitrarily many uploaded photos.
- **Blast radius**: 2 files, one-attribute fix each.
- **Behaviour-change risk**: LOW — both grids already use fixed-size containers, no layout-shift risk.

### S11. 115 of 116 interactive `div`/`span` elements with `onClick` have no keyboard support (broader tail of B2)
- **File(s)**: Highest concentration `WBLeftPanel.jsx` (6), `AdvancedInvitationBuilder.jsx` (6, dead code per B4 — no fix needed there), `MediaLibraryModal.jsx` (4), `MoodboardGrid.jsx` (4), plus ~50 more files with 1-3 each — including guest-facing surfaces: `WeddingPhotosPage.jsx:33` (guest photo lightbox), `GuestMusic.jsx:159` (guest song rows), `EntranceMoment.jsx:178` (partially fixed — has `role`/`aria-label` but still no `tabIndex`/`onKeyDown`)
- **Evidence**: Only one instance in the whole app (`Checklist.jsx:73`) does this correctly.
- **Blast radius**: Systemic across both dashboard and public guest-facing pages.
- **Behaviour-change risk**: LOW — additive `role="button" tabIndex={0} onKeyDown={...}`.

### S12. ~55 icon-only modal close/delete buttons have no `aria-label`; Escape-key fallback exists in only 5 files
- **File(s)**: Representative: `MusicSuggestionsModal.jsx:172`, `VendorDetailPanel.jsx:153`, `RegistryForm.jsx:20`, `AddTableModal.jsx:26`, `BudgetForm.jsx:49`, `GuestForm.jsx:139`, `SendInvitesModal.jsx:542`, `WeddingParty.jsx:222` — 61 total icon-only buttons with no label, spanning nearly every dashboard feature area

- **Behaviour-change risk**: LOW — additive.

### S13. CLAUDE.md's own mandated muted-text color fails WCAG AA contrast by a wide margin
- **File(s)**: 504 uses of `rgba(10,10,10,0.4)` as `color`, incl. guest-facing `RSVPPage.jsx:59,686`, `TodoList.jsx:215,220`, `GuestList.jsx:143,259`
- **Evidence**: Computed contrast ≈2.7:1 against white (needs 4.5:1 for normal text, 3:1 even for large/bold text) — fails both.
- **Blast radius**: The single largest-blast-radius a11y finding — this is the *documented, sanctioned* convention, not a mistake, used across both dashboard and guest-facing surfaces.
- **Behaviour-change risk**: **Needs your sign-off before any fix.** CLAUDE.md explicitly mandates this exact value and separately forbids `#888`/`gray-400/500` — but the mandated value itself doesn't meet accessibility minimums. Darkening it is a foundational design-token change repeated 500+ times, not a one-off — flagging rather than proposing a specific replacement value, since that's a design decision.

### S14. An undocumented, even-lighter muted-text variant is also used for real text
- **File(s)**: 76 uses of `rgba(10,10,10,0.3)`, incl. `RSVPPage.jsx:59,686`, `TodoList.jsx:215,220,358`, `GuestList.jsx:143,259,503`
- **Evidence**: Contrast ~2.7:1 or below — same failure class as S13, but not even the documented token.
- **Blast radius**: ~25+ files.
- **Behaviour-change risk**: LOW-MEDIUM — since this isn't the mandated token, it can be corrected independently of the S13 decision (though ideally standardized onto whatever S13 lands on).

### S15. `gray-400`/`gray-500`/`#888`/`#999`/`#aaa` used despite CLAUDE.md explicitly forbidding them
- **File(s)**: ~90 files (392 combined grep hits) — e.g. `RSVPChart.jsx`, `website-builder/*`, `studio/*`, `About.jsx`, `Contact.jsx`, `EventsSection.jsx`, `CollaborateModal.jsx`
- **Evidence**: `#888` ≈3.55:1 (fails AA), Tailwind `gray-400` ≈2.87:1 (fails AA), `gray-500` ≈4.83:1 (passes narrowly) — a second, competing "muted gray" convention, independent of S13/S14, itself also partly contrast-failing.
- **Blast radius**: ~90 files, systemic design-system drift.
- **Behaviour-change risk**: LOW to migrate onto the documented token (that's literally the existing rule) — but do this *together* with the S13 contrast fix, not before it, or everything just moves onto an equally-failing color.

### S16. White headline text on the homepage gradient banner is functionally invisible
- **File(s)**: `src/pages/Home.jsx:76-82`

- **Blast radius**: One instance, but the largest/most prominent text on the homepage.
- **Behaviour-change risk**: LOW to fix locally (text-shadow/scrim, or a solid dark color for that segment) — but the gradient itself is presumably an intentional brand choice, so don't recolor the gradient, just the text treatment.

## Positive findings (no action needed)

## Proposed fix sequence — safest first

Each chunk is sized to be one PR. Chunks are ordered so the safest, most mechanical work lands first and the items needing your explicit decision are pulled out and sequenced last, not blocking anything else.

1. **Dead code removal** (B4) — delete all 78 confirmed-dead component files. Zero behaviour risk (double-verified). Biggest line-count win in the audit for the least risk.
2. **Unused dependency cleanup** (S23, N6, N7) — remove the 9 confirmed-unused prod deps + 1 devDependency; `npm run build` after. Lower-confidence ones (N7) get a `npm ls` check first, not blind removal.
3. **Accessibility: global chrome + sidebar** (B2, B3) — keyboard support on `AnimatedSidebar`'s `NavItem`, `aria-label` on the bell/hamburger/Ava-toggle. Small, isolated, highest a11y impact per line changed.
4. **Accessibility: long tail** (S11, S12, S17, S18) — keyboard support on the remaining ~115 onClick divs, aria-labels on ~55 close/delete buttons, the 2 missing-alt images, and `alt=""` → descriptive alt on informative images. Larger file count but mechanically identical, low-risk changes — worth its own PR separate from #3 given the size.
5. **Performance: safe additive fixes** (S9, S10, N2) — `Promise.all` in `collaborator-data.js`, `loading="lazy"` across photo/moodboard/marketplace/guest-website images (verify fixed dimensions first).
6. **Data flow: safe additive fixes** (S1, S2, S7) — checkout error surfacing in `PlanSelection.jsx`, Dashboard error toast, loading states on Budget/Schedule/Vendors. All modeled on patterns already shipped elsewhere in this codebase.
7. **Memory: timeout cleanup** (N4) — `OnboardingCompletion.jsx` first (real risk), then the two sub-20ms ones. Small, isolated.
8. **Guests/GuestList memoization** (S8) — needs its own PR with explicit test coverage of inline-edit/bulk-select/quick-add afterward, since incorrect `useCallback`/`useMemo` deps risk stale-closure bugs.
9. **Bundle/code-splitting** (B1) — `React.lazy` for `App.jsx` routes + dynamic `import()` for xlsx/jspdf/html2canvas. Biggest performance win in the audit; needs a deliberate `<Suspense>` fallback design, so give it care and its own review pass rather than rushing it.
10. **Data-flow consolidation** (S3, S4, S5) — caching layer adoption, `Layout.jsx` refetch-trigger review, `useCollaboratorContext` → shared Context. Bigger architectural change; S4 specifically needs a decision on whether the existing `weddingDetailsSaved` event already covers invalidation before removing the `location.pathname` trigger.
11. **`api/places.js` status-contract fix** (S6) — isolate this from #10 even though it's data-flow-adjacent, since it specifically risks breaking an existing "always 200" client assumption if not scoped carefully to only the error branches.
12. **Muted-text contrast fix** (S13, S14, S15, S16) — **needs your decision first**, not a rubber-stamp PR: what should the corrected muted-text value be, given CLAUDE.md currently mandates the failing one. Do S13/S14/S15 together once decided, so everything lands on the same corrected token in one pass. S16 (homepage banner) can go in the same PR since it's the same root problem (light/white text on a light background).
13. **RSVP-tally consolidation** (S21) — **needs your decision first**: which status convention is correct (`'attending'` alone vs. `'confirmed'`-or-`'attending'`)? The counts may already silently disagree between screens today — worth checking live data before deciding, since this could be an existing user-facing bug, not just a duplication risk.
14. **Routing consolidation** (S20) — **needs your decision first**: canonical URL casing per affected page, and confirmation that removing the PascalCase duplicates won't break any external links/bookmarks.
15. **Legacy page removal** (S22) — confirm `CouplesStudio.jsx`/`WebsiteCustomization.jsx` aren't linked externally before deleting (lower stakes than #13/#14, but still a "confirm first" item since they're still routable, unlike #1's fully-dead files).
16. **Mobile/responsive messaging** (S19, N9) — lowest priority given both are desktop-oriented power-user features by nature; add "best viewed on desktop" messaging as a small, separate PR whenever convenient.

Items **12, 13, and 14 are the only ones in this whole audit that need your explicit decision before any code is written** — everything else is either safe-to-execute-as-described or has enough evidence to proceed once scheduled.
